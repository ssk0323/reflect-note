---
name: spec
description: 個別 Issue の受け入れ条件や仕様を着手前にブラッシュアップする。/plan で起票された Issue が曖昧なときや、TDD 着手直前に受け入れ条件を観測可能な形に整えたいときに使う。
---

**前提**: 全体像 (`docs/overview.md`) と計画 (`docs/plan.md`) は既に存在する想定。
新規プロジェクト・新領域の最初は `/overview` から始めること。

個別 Issue の仕様を、TDD で実装可能なレベルまで深掘りする。

# 手順

1. **対象 Issue を特定**
   - `$ARGUMENTS` に Issue 番号があればそれを使う
   - なければ「どの Issue を詰めますか？」と確認
   - `gh issue view <番号>` で現状の受け入れ条件・スコープを取得

2. **足りない情報を洗い出す**
   - 受け入れ条件は **観測可能・検証可能** か？
   - エッジケース（空、エラー、未認証、競合状態など）が漏れていないか？
   - UI なら画面要素・操作フローが具体的か？
   - データなら入力形式・必須/任意・バリデーションが具体的か？

3. **spec-interviewer agent を起動**
   - Agent ツールで `subagent_type: spec-interviewer`
   - prompt に Issue URL と「補強したい点」を渡す
   - サブエージェント側で `docs/overview.md` を必ず読ませる（既存設計との整合性のため）

4. **ユーザーと擦り合わせ**
   - agent が返した補強案を表示
   - 必要に応じて AskUserQuestion で詰める

5. **Issue を更新**
   - 合意できた内容で `gh issue edit <番号> --body "..."` する（または PR 用にローカルメモとして残す）
   - 重要な決定は Issue コメントにも追記して履歴化

6. **次のステップを促す**
   - 「`/tdd <番号>` で実装に入りますか？」

# 注意

- ここで **新機能を追加しない**。スコープ拡大したくなったら `docs/plan.md` の更新を提案
- 既に `docs/overview.md` に書かれていることを聞き直さない
- ヒアリングしすぎない。受け入れ条件が観測可能になったら止める
