アプリ全体の "全体像ドキュメント" を作成する。これがその後の `/plan`・`/issue`・実装の基準点になる。

# 手順

1. **既存情報を集める**
   - `CLAUDE.md` を必ず読む
   - 既存の `docs/overview.md` があれば読む（更新モード）
   - `$ARGUMENTS` でファイルパスが指定されていれば中身を読む（要望メモ、既存仕様書、モックなど）
   - リポジトリ直下を `ls` して、現在のコード状況を把握

2. **spec-interviewer agent を呼ぶ**
   - prompt には「ユーザーから全体像をヒアリングしてほしい」「対象資料の絶対パス」「成果物は `docs/overview.md` のドラフト」と明示
   - サブエージェントは AskUserQuestion を使えないので、推測ベースのドラフトを返してもらう

3. **ユーザーと擦り合わせる**
   - agent のドラフトを表示
   - 不明点を AskUserQuestion で確認（技術選定、優先機能、スコープ外、データの保存先、認証方式、デプロイ先 など）
   - 合意できた内容で `docs/overview.md` に書き出す

4. **`docs/overview.md` を保存**
   - 既存ファイルがあれば Edit、なければ Write
   - 以下のセクション構成を推奨:

```markdown
# reflect-note 全体像

## 1. 目的とコンセプト
（1〜2 段落で「なぜ作るか」「誰のためか」）

## 2. 想定ユーザー / 利用シーン

## 3. 主要機能（MVPスコープ）
- ...
- ...

## 4. 将来の拡張候補（MVPに含めない）
- ...

## 5. データモデル（概要）
（主要なエンティティと関係。詳細は別 Issue で）

## 6. 技術構成
- フロント: ...
- DB: ...
- 認証: ...
- デプロイ先: ...
- テスト: ...

## 7. 非機能要件
- パフォーマンス / アクセシビリティ / セキュリティ / プライバシー

## 8. スコープ外（明示的にやらないこと）

## 9. オープン課題 / 未確定事項
```

5. **次のステップを促す**
   - 「`/plan` でこの全体像を実装計画に分解しますか？」

# 注意

- 全体像は **生きたドキュメント**。後で更新されることを前提に、断言しすぎず根拠を書く
- 細かい受け入れ条件はここでは書かない（それは `/spec` の仕事）
- ユーザーが既に書いた仕様書（例: `.temp/*.md`）がある場合は、それを引用元として明記
- 機能名・データモデル名はここで確定させる（後の Issue で表記揺れしないように）
