## 1. 目的とコンセプト

reflect-note は、**朝・夜・週・月の単位で「目標設定」と「振り返り」を一問一答で短く記録する** ためのリフレクション支援 Web アプリ。

- 「朝に1日の目標とタスクを整え、夜に1日を振り返る。さらに、週次・月次の目標設定と振り返りを行うことで、日々の行動を中長期の目標につなげる」（ChatGPT 製 MVP 仕様書 1.2 より）
- 開発者自身が PC とスマホの両方で気軽に使うためのツールで、**継続できることを最優先** に設計する
- 個人利用が前提のため、**他人に内容を見られない** ことを必須要件とする

---

## 2. 想定ユーザー / 利用シーン

### 想定ユーザー
- 日々の振り返りを習慣化したい人
- 1日の目標やタスクを整理したい人
- 不安やモヤモヤを簡単に言語化したい人
- 週次・月次で自分の行動や成果を振り返りたい人

当面は **開発者本人が唯一のユーザー**。Google ログインは自分のアカウントのみ許可リストで縛る（後述）。

### 利用シーン
- **朝**: 1日の目標とタスクを決める
- **夜**: 1日を振り返り、明日につなげる
- **週初め / 週末**: 今週の目標 / 振り返り
- **月初 / 月末**: 今月の目標 / 振り返り

PC（自宅・仕事の合間）とスマホ（移動中・寝る前）の両方からアクセスする想定。

## 3. 主要機能（MVP スコープ）

機能名はモック JSX・MVP 仕様書の表記に合わせる。

| カテゴリ | 機能 | 概要 |
|---|---|---|
| 入力フロー | 朝のセットアップ (`morning`) | 今日の目標・タスク3つ・気をつけたいこと |
| 入力フロー | 夜のリフレクション (`night`) | できたこと / よかったこと / モヤモヤ / 明日に回すこと / Try / 時間の使い方（朝・午前・午後・夜） / 明日の自分へひとこと |
| 入力フロー | 週の目標設定 (`weeklyGoal`) | 今週の目標・大事にしたいこと・優先タスク3つ |
| 入力フロー | 週の振り返り (`weeklyReview`) | 今週できたこと・よかったこと・学び・来週のTry |
| 入力フロー | 月の目標設定 (`monthlyGoal`) | 今月の目標・テーマ・重点タスク3つ |
| 入力フロー | 月の振り返り (`monthlyReview`) | 今月できたこと・よかったこと・学び・来月のTry |
| 入力 UI | 一問一答入力 | 1画面1質問。進捗バー表示。戻る/次へ。夜の「時間の使い方」のみ1画面4項目 |
| 入力 UI | 入力内容確認 | 保存前に一覧表示。修正 / 保存 |
| 入力 UI | 下書き一時保持 | 入力途中の内容を localStorage に保持し、離脱後の復元を可能にする |
| 表示 | トップ画面 | 今週の目標・今月の目標・各入力フロー開始カード・最新記録 |
| 表示 | 過去記録一覧 | 日付グルーピング / 種別ラベル / フィルタ（朝・夜・週目標・週振り返り・月目標・月振り返り） |
| 表示 | 記録詳細 | アコーディオン展開で内容確認 |
| 編集 | 記録編集 | 既存内容を初期値に再入力。`updatedAt` 更新 |
| 編集 | 記録削除 | 確認ダイアログ → 削除 |
| 認証 | Google ログイン | Supabase Auth + Google OAuth。自分のメールアドレスのみ許可 |
| 同期 | クラウド保存 | Supabase Postgres、RLS で行レベル分離 |
| ゲーミフィケーション | ストリーク | 朝/夜の連続記録日数を表示 |
| ゲーミフィケーション | 週次/月次の入力完了バッジ | その週・月に必要な入力を達成したらバッジ付与 |
| ゲーミフィケーション | 月間達成バッジ | 月の入力回数しきい値（例: 20回）でバッジ付与 |

### MVP 仕様書からの差分（重要）
- **localStorage 単独の保存は採用しない**（記録本体は Supabase に保存。同期可能にする）
- **認証を MVP に含める**（他人に見られない要件）
- **ゲーミフィケーション（ストリーク / バッジ）を MVP に含める**

## 5. データモデル（概要）

モック JSX のレコード構造（`id` / `type` / `createdAt` / `updatedAt` / `answers`）をベースに、Supabase + マルチユーザー対応に拡張する。

### 5.1 `users`
Supabase Auth が管理。`auth.users.id` (UUID) を他テーブルの `user_id` 外部キーとして参照。プロフィール拡張は MVP では不要。

### 5.2 `records`
| カラム | 型 | 説明 |
|---|---|---|
| `id` | uuid (PK) | レコード ID |
| `user_id` | uuid (FK → auth.users.id) | 所有ユーザー。RLS のキー |
| `type` | text | `morning` / `night` / `weeklyGoal` / `weeklyReview` / `monthlyGoal` / `monthlyReview` |
| `answers` | jsonb | 質問キー → 回答文字列。スキーマはアプリ側 (`FLOWS`) で管理 |
| `created_at` | timestamptz | 作成日時。今日/今週/今月判定に使う |
| `updated_at` | timestamptz | 更新日時 |

#### インデックス
- `(user_id, created_at DESC)` — 一覧の高速化
- `(user_id, type, created_at DESC)` — フィルタ + 「今週/今月の目標」検索

#### RLS ポリシー
- `select` / `insert` / `update` / `delete` を `user_id = auth.uid()` のみ許可

#### 派生キー（アプリ層で算出）
- `weekKey` = 月曜始まりの ISO 日付（モック JSX `getWeekKey` を移植）
- `monthKey` = `YYYY-MM`

### 5.3 ゲーミフィケーション関連

| テーブル | 用途 |
|---|---|
| `streaks` | `user_id`, `kind` (`morning` / `night` など), `current_count`, `longest_count`, `last_recorded_on` |
| `achievements` | `user_id`, `code` (例: `weekly_complete_2026W21`, `monthly_20_2026_05`), `unlocked_at` |

ストリークとバッジは「records が増えたタイミングでアプリ側 or Supabase の trigger で再計算」する。実装方式（クライアント側計算 / DB トリガー / Edge Function）は Issue で詰める。

### 5.4 下書き
記録本体は Supabase に保存するが、**入力途中の未保存データは localStorage に一時保持** する。フロー単位（`morning`, `night`...）でキーを分け、保存 / 破棄時にクリアする。

## 7. 非機能要件

### 7.1 レスポンシブ
- **PC とスマホの両方で快適に使える**こと（必須）
- 入力画面はスマホで使いやすい縦長レイアウトを基本（MVP 仕様書 8.2 を踏襲）
- ブレークポイントはモック JSX 準拠（`sm:` / `md:` / `lg:`）

### 7.2 パフォーマンス
- 初回ロード（認証チェック含む）でトップ画面表示まで概ね 2 秒以内を目標
- 入力フロー内のページ遷移は即時（楽観的更新を許容）

### 7.3 セキュリティ / プライバシー
- **他人に記録を見られない** ことが必須要件
- Supabase RLS で行レベル分離
- Google OAuth で **自分のメールアドレスのみ許可リスト** で縛る（実装方式は Issue で詰める）
- 環境変数（`.env.local`）で API キー管理、`.gitignore` 必須
- `SUPABASE_SERVICE_ROLE_KEY` は **クライアントに出さない**
- Google OAuth リダイレクト URL は Vercel 本番ドメインを登録

### 7.4 アクセシビリティ（最低限）
- フォーム要素に `<label>` 紐付け
- フォーカスリング保持（Tailwind の `focus:ring-*`）
- カラーコントラスト基準 WCAG AA 程度を目標

### 7.5 対応ブラウザ
- Chrome / Safari / Edge の最新版

## 9. オープン課題 / 未確定事項

### 認証
- 「自分のメールアドレスのみ許可」の実装方式: Supabase Auth ポリシー / DB トリガーで `auth.users` を見て弾く / 専用 middleware で許可リスト検証、のどれか。Issue で決める

### ゲーミフィケーション
- ストリーク・バッジの再計算方式: クライアント側計算 / DB トリガー / Edge Function のいずれか
- バッジ条件の具体的なしきい値（月20回 など）
- ストリーク表示の置き場所（トップ画面のどこに出すか）

### インフラ / 運用
- Vercel / Supabase のリージョン選定（Tokyo / `ap-northeast-1` を想定）
- バックアップ方針（Supabase Free プランの自動バックアップで十分か）

### データモデル
- 質問キー (`FLOWS`) を変更した場合の既存レコードへの影響と移行方針
- 編集履歴（version 管理）は MVP では持たない想定

### UI / UX
- ストリーク・バッジ獲得時のフィードバック演出（祝う UI を入れるか）
- 一覧の絞り込み UI のスマホ対応（モックは横並びだが収まりが悪い可能性）
