全体像ドキュメント (`docs/overview.md`) から、実装可能な計画（マイルストーン + Issue 候補）に分解する。

# 手順

1. **前提を確認**
   - `docs/overview.md` を読む。なければ「先に `/overview` を実行してください」と促す
   - `CLAUDE.md` の開発フロー・命名規則・ラベル定義を確認
   - 現在の GitHub Issue リストを `gh issue list --state all` で取得（既存と重複しないように）

2. **マイルストーンを切る**
   - 例: M0 セットアップ / M1 認証 / M2 MVP コア機能 / M3 ゲーミフィケーション / M4 拡張機能
   - 各マイルストーンに「完了条件」（観測可能な状態）を 1〜2 行で

3. **Issue 候補を起こす**
   - 1 Issue = 1 PR で完結する粒度に切る
   - 各 Issue に以下を含める:
     - 仮タイトル（Conventional Commits 形式: `feat: ...` `chore: ...`）
     - 一言での目的
     - 主な受け入れ条件（3〜5 個。これは Issue 起票時に深掘りされる）
     - ラベル候補（`type:*`, `priority:*`）
     - 依存関係（先にやるべき Issue があれば）
   - 大きすぎる Issue は分割。受け入れ条件が 7 個を超えたら割る

4. **ユーザーと擦り合わせる**
   - ドラフトを表示し、AskUserQuestion で「優先順位」「分割の妥当性」「外したい項目」を確認

5. **`docs/plan.md` を保存**
   - 構成:

```markdown
# reflect-note 実装計画

最終更新: YYYY-MM-DD

## マイルストーン

### M0: ...
完了条件: ...
- [ ] #N? feat: ...
- [ ] ...

### M1: ...
完了条件: ...
- [ ] ...

## Issue 候補（詳細）

### feat: ...
- 目的:
- 主な受け入れ条件:
  - ...
- ラベル: type:feat, priority:high
- 依存: #N（先にやる）

...
```

6. **次のステップを促す**
   - 「`/issue` で M0（または特定のマイルストーン）の Issue をまとめて起票しますか？」
   - 一括起票したい範囲をユーザーに選ばせる

# 注意

- 計画も **生きたドキュメント**。実装中に分割・統合・並べ替えが起きる前提でゆとりを持って書く
- 1 マイルストーンの中で Issue が 10 個を超えたら分割を検討
- 「やらないこと」が overview にあれば、Issue 候補にも入れない
- 依存関係は最小限に。直列にしすぎると進捗が詰まる
